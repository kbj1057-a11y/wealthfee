# [SOP] 월간 수입수수료 통합 및 검증 표준 운영 절차

이 문서는 매달 27일 발생하는 수입수수료 정산 작업을 실수가 없도록 자동화하고 관리하기 위한 표준 절차(SOP)입니다.

## 1. 데이터 준비 (Preparation)
- **대상 폴더**: `.../27일수수료자료/[YY]년수수료자료/[YYMM]수입수수료산출`
- **입력 파일**:
    1. **보험사 원본**: `(사업부)수입수수료산출_*.xlsx` (다수)
    2. **이파트너 업데이트**: `[YYMM]이파트너수수료업데이트.xlsx` (1개)

## 2. 1단계: 보험사 데이터 통합 (Merging)
- **분류 로직**: 파일명이 아닌 **데이터 내 '제휴사명'**을 기준으로 생명/손보 시트로 분리.
- **헤더 표준화**:
    - **J열(Index 9)**: '사번'으로 고정.
    - **K열(Index 10)**: 'FC명'으로 고정.
    - **25열(실적)**: 생명('환산실적'), 손보('수정보험료')로 명칭 최적화.
    - **원처**: 원본 파일명 정보는 삭제하여 가독성 확보.

## 3. 2단계: 이파트너 데이터 보완 (Enrichment)
- **매칭 키 (Composite Key)**: 반드시 **(증권번호 + 정규화된 제휴사명)**을 결합하여 매칭.
    - 이유: 서로 다른 보험사에서 동일한 증권번호를 사용할 가능성 차단.
- **데이터 정규화 (CRITICAL)**:
    - **증권번호**: 숫자 뒤에 `.0`이 붙는 현상을 제거(문자열 처리)하여 매칭률 100% 확보.
    - **제휴사명**: '화재', '생명' 등의 접미사를 제거한 '순수 사명'으로 매칭.
- **중복 계산 방지**: 
    - 보험사 원본 데이터에 동일 증권번호가 여러 행일 경우, 이파트너 수수료는 **'첫 번째 행'에만 배부**하고 나머지는 0으로 처리하여 전체 합계가 부풀려지는 것을 방지.
- **데이터 최적화 (Row Filtering)**:
    - `최종수수료`가 **0**인 행은 삭제하여 `data.js` 크기를 경량화함.
    - 단, 마이너스(-) 금액(환수 건)이나 e-Partner 매칭 수수료가 있는 행은 보존함.
- **대시보드 데이터 송출 (Data Export)**:
    - `scripts/export_to_data_js.py`를 통해 최종 데이터를 `data.js`로 변환.
    - 변환 시 `index.html`이 인식할 수 있는 컬럼명(환산성적, 보험료 등)으로 매핑함.

## 4. 3단계: 최종 검증 (Reconciliation)
- **검증 기준**: 마스터 파일의 `지사수수료` 합계가 사용자(이파트너)의 검증 수치와 정확히 일치해야 함.
- **체크리스트**:
    - [ ] 생명보험 시트 지사수수료 합계 확인
    - [ ] 손해보험 시트 지사수수료 합계 확인
    - [ ] 증권번호 형식(String) 및 중복 매핑 여부 확인

## 5. 과거 변수 및 해결 사례 (Lessons Learned)
- **파일명 인코딩**: G 드라이브 환경에서 한글 깨짐 대응 -> 내부 데이터(컬럼) 기반 로직 사용.
- **파일 권한**: 엑셀이 열려 있을 때 발생하는 `Permission Error` -> 출력 파일명 버전 관리(`_final`, `_verified` 등).
- **합계 행**: 원본 하단의 '합계' 행이 데이터에 포함되지 않도록 필터링(`df['NO'] != '합계'`).
